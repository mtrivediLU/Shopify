Technical thinking

How I would approach a merchant integration.

A simplified view of how I would frame an integration between a merchant system and the Shopify Admin API. Drawn from general integration experience, not from any internal Shopify knowledge.

  1. 01

    Merchant system or ERP

    Orders, inventory, customers, or product data living in the merchant's source of truth.

  2. 02

    Integration service

    A small backend service that owns the contract between the merchant system and Shopify.

  3. 03

    Authentication and API boundary

    Scoped tokens, secret rotation, and a clear API surface that hides upstream variability.

  4. 04

    Webhook ingestion

    Signed webhook handlers that acknowledge fast and hand work to background processors.

  5. 05

    Validation and transformation

    Strict schema checks, normalization, and clear error shapes before anything touches the database.

  6. 06

    Database persistence

    PostgreSQL with thoughtful indexing, foreign keys where they help, and migrations that are safe to roll forward.

  7. 07

    Retry and dead letter handling

    Bounded retries with exponential backoff, plus a dead letter queue for human review.

  8. 08

    Shopify Admin API layer

    Calls to the relevant commerce API behind a thin client that owns auth, retries, and rate limits.

  9. 09

    Observability and alerting

    Structured logs, traces, business metrics, and alerts tied to merchant facing impact.

  10. 10

    Merchant facing status view

    A simple status surface so merchants see what synced, what failed, and what is in flight.

Engineering practices I would lean on

Idempotency

Stable external IDs and idempotency keys so retries do not double create orders or inventory updates.

Rate limit aware API calls

Respect documented limits, read response headers, and use queue backoff before the API pushes back.

Webhook verification

Verify signatures, reject replays with timestamp checks, and log raw payloads for replay.

Background jobs

Move anything slow or risky off the request path. Keep webhook handlers thin and fast.

Error recovery

Classify errors as retryable or terminal, surface terminal failures clearly, and avoid silent drops.

Audit logging

Record who or what changed each record, with enough context to answer a merchant question without guessing.

Test coverage

Unit tests on transforms, contract tests on API clients, and a handful of integration tests on the happy and failure paths.

Production monitoring

Dashboards tied to merchant impact, runbooks for the top failure modes, and alerts that actually mean something.
