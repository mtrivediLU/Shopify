How I think

Before code, I try to understand the merchant outcome.

Start with the operating problem

In integration work, the difficult part is often not the endpoint itself. It is the unclear business rule, the external system that behaves slightly differently than expected, the missing edge case, and the launch timeline that still matters to the merchant. My first step is to clarify the actual outcome, not simply the requested implementation.

Design for failure from the beginning

Webhooks can be duplicated. APIs can rate limit. Jobs can fail halfway. Data can arrive late. A production integration should expect those realities. I think about idempotency, validation, retries, audit trails, observability, and rollback before I think about making the happy path look elegant.

Build trust quickly, then compound it

In client facing engineering, the trust battery starts from zero. I try to build it with clear communication, small working increments, early risk surfacing, readable code, and documentation that helps the next person maintain the system. That is how I would approach merchant facing work at Shopify.

Craft demo

A small Rails example showing how I think about webhooks.

I am preparing for the pair programming round by practicing Ruby on Rails patterns, especially the parts that matter in integrations: verification, idempotency, background job offload, clear responses, and production friendly failure handling.

Ruby on Rails Webhook controller sketch 
class Webhooks::OrdersController < ApplicationController
  skip_before_action :verify_authenticity_token

  def create
    payload = request.raw_post
    signature = request.headers["X-Shopify-Hmac-Sha256"]

    unless valid_hmac?(payload, signature)
      Rails.logger.warn("Rejected webhook with invalid signature")
      return head :unauthorized
    end

    event_id = request.headers["X-Shopify-Event-Id"]
    topic = request.headers["X-Shopify-Topic"]

    webhook_event = WebhookEvent.find_or_initialize_by(
      external_event_id: event_id
    )

    if webhook_event.persisted?
      Rails.logger.info("Duplicate webhook ignored: #{event_id}")
      return head :ok
    end

    webhook_event.assign_attributes(
      topic: topic,
      payload: JSON.parse(payload),
      status: "queued"
    )

    webhook_event.save!
    ProcessOrderWebhookJob.perform_later(webhook_event.id)

    head :accepted
  rescue JSON::ParserError
    Rails.logger.warn("Webhook rejected because payload was invalid JSON")
    head :bad_request
  rescue ActiveRecord::RecordInvalid => error
    Rails.logger.error("Webhook persistence failed: #{error.message}")
    head :unprocessable_entity
  end

  private

  def valid_hmac?(payload, signature)
    digest = OpenSSL::HMAC.digest(
      "sha256",
      Rails.application.credentials.shopify_webhook_secret,
      payload
    )

    expected = Base64.strict_encode64(digest)

    ActiveSupport::SecurityUtils.secure_compare(
      expected,
      signature.to_s
    )
  end
end
01

Verify before trusting

HMAC verification happens before parsing or persistence, so untrusted payloads are rejected early.

02

Make duplicate delivery safe

The external event id becomes the idempotency boundary. Duplicate webhook deliveries return safely.

03

Offload work to a job

The request path stores the event and returns quickly. Heavy processing runs asynchronously.

04

Log what helps production

The code records enough signal for debugging without exposing sensitive payload details in logs.

Integration architecture

How I would approach a merchant integration.

This is a simplified architecture view, not a claim about Shopify internal systems. It shows the questions I would ask and the reliability patterns I would consider.

Merchant system or ERP

Webhook ingestion
Validation and transformation

Commerce API layer
Observability and alerts
Merchant status view

Idempotency

Every external event should have a safe replay story so duplicate delivery does not create duplicate business impact.

Rate limit awareness

API clients should use backoff, batching where appropriate, and clear retry rules instead of forcing fragile traffic patterns.

Error recovery

Failures should be visible, classified, and recoverable through retries, dead letter handling, or a manual support path.

Audit logging

Integration events should leave enough history to explain what happened, when it happened, and what changed.

Test coverage

Tests should cover signature verification, duplicate delivery, invalid payloads, retry behavior, and happy path processing.

Production monitoring

Operational dashboards should surface queue depth, job failure rate, webhook volume, API errors, and merchant impact.
